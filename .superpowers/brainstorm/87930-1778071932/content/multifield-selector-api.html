MultiFieldSelector API Design

How should MultiFieldSelector manage the array state?

A

Self-Contained (Simple API)

MultiFieldSelector owns the array logic internally. Parent just passes value/onChange for the whole array.

// Usage in LineChartSettings
<MultiFieldSelector
label="Y Axes"
columns={columns}
types={['INTEGER', 'DOUBLE']}
value={values.yFields}
onChange={(yFields) => updateField('yFields', yFields)}
showAggregation={Boolean(values.xInterval)}
/>
// Internal implementation
function MultiFieldSelector({value, onChange, ...}) {
const handleUpdate = (index, updates) => {
const updated = [...value];
updated[index] = {...value[index], ...updates};
onChange(updated);
};

const handleRemove = (index) => {
onChange(value.filter((_, i) => i !== index));
};

const handleAdd = (field) => {
onChange([...value, {field, aggregate: 'sum'}]);
};

return (/* render rows */);
}

Pros

  • Clean parent API - just value/onChange
  • Encapsulates array logic
  • Reusable across chart types
  • Matches existing MultiFieldSelector

Cons

  • Less control for custom layouts
  • Hard to customize individual rows
B

Render Props / Children (Flexible)

MultiFieldSelector provides state helpers, parent renders the actual UI.

// Usage in LineChartSettings
<MultiFieldSelector
label="Y Axes"
value={values.yFields}
onChange={(yFields) => updateField('yFields', yFields)}
>
{({fields, updateField, removeField, addField}) => (
<>
{fields.map((field, index) => (
<MultiFieldSelector.Row key={field.field}>
<ColumnSelector
columns={columns}
value={field.field}
onChange={(f) => updateField(index, {field: f})}
/>
{values.xInterval && <AggregationSelector
value={field.aggregate}
onChange={(agg) => updateField(index, {aggregate: agg})}
/>}
<DeleteButton onClick={() => removeField(index)} />
</MultiFieldSelector.Row>
))}
<MultiFieldSelector.AddRow
columns={columns}
onAdd={(field) => addField({field, aggregate: 'sum'})}
/>
</>
)}
</MultiFieldSelector>

Pros

  • Full control over row rendering
  • Easy to add custom UI per chart
  • Conditional rendering (aggregation)

Cons

  • More verbose usage
  • Pushes layout to parent
  • Harder to ensure consistency
C

Hybrid: Default UI + Custom Slots

MultiFieldSelector renders default UI but accepts custom components via slots/props.

// Simple usage (default UI)
<MultiFieldSelector
label="Y Axes"
columns={columns}
types={['INTEGER', 'DOUBLE']}
value={values.yFields}
onChange={(yFields) => updateField('yFields', yFields)}
showAggregation={Boolean(values.xInterval)}
/>

// Custom usage (with slots)
<MultiFieldSelector
label="Y Axes"
value={values.yFields}
onChange={(yFields) => updateField('yFields', yFields)}
renderField={(field, index, {updateField, removeField}) => (
<>
<ColumnSelector ... />
<AggregationSelector ... />
<ColorPicker {/* custom! */}
value={field.color}
onChange={(c) => updateField(index, {color: c})}
/>
<DeleteButton onClick={() => removeField(index)} />
</>
)}
/>

Pros

  • Simple by default, flexible when needed
  • Reusable for common cases
  • Customizable for special cases

Cons

  • More complex implementation
  • Two APIs to maintain

Context for Decision

Current code: You already have a working MultiFieldSelector (Option A style) that internally manages array updates.

Your concern: Flexibility - each chart type might need different controls per row.

Question: Do you envision chart-specific customizations (like color pickers, custom validators) or will most charts use the same column + aggregation pattern?
